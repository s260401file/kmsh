護理白板系統 — IIS 部署說明

適用版本：React 19 + Vite 8  |  ASP.NET Core .NET 8  |  Windows Server IIS
最後更新：2026-05-30

系統架構

部署完成後，院內同仁透過內網 IP 訪問，整體架構如下：

【院內同仁瀏覽器】
       │
       │  http://<伺服器內網IP>  (port 80)
       ▼
【IIS — 前端網站 kmsh-whiteboard】
       │
       ├─ 靜態檔案 /        → C:\inetpub\wwwroot\kmsh-whiteboard\  (React build)
       ├─ /api/*  ──ARR──▶  http://localhost:5100/api/*            (轉發到 API)
       └─ 其他路徑           → /index.html                          (SPA fallback)
       │
       ▼
【IIS — API 網站 kmsh-whiteboard-api】
       監聽 127.0.0.1:5100  (僅本機，不對外)
       實體路徑：C:\inetpub\wwwroot\kmsh-api\
       .NET 8 in-process hosting
元件網址實體路徑
React 前端http://<內網IP>/C:\inetpub\wwwroot\kmsh-whiteboard\
.NET APIhttp://localhost:5100/（僅本機）C:\inetpub\wwwroot\kmsh-api\

必要模組安裝

以下模組若未安裝，依序完成後再繼續部署步驟。

模組 確認方式 下載位置
.NET 8 Hosting Bundle
讓 IIS 執行 ASP.NET Core		 IIS 管理員頂層節點 → 模組 → 看到 AspNetCoreModuleV2 dotnet.microsoft.com/download/dotnet/8.0
→ ASP.NET Core Runtime → Hosting Bundle
（檔名：dotnet-hosting-8.x.x-win.exe）
URL Rewrite
SPA 路由 fallback 規則		 IIS 管理員首頁看到「URL 重寫」圖示 iis.net/downloads/microsoft/url-rewrite
（檔名：rewrite_amd64_en-US.msi）
ARR 3.0
將 /api/* 反向代理到本機 API		 IIS 管理員伺服器頂層節點首頁看到「Application Request Routing Cache」圖示
⚠️ 注意：此圖示只出現在頂層伺服器節點，不在網站節點		 iis.net/downloads/microsoft/application-request-routing
（檔名：requestRouter_amd64.msi，約 9 MB）
安裝後重新開啟 IIS 管理員才會看到圖示
⚠️
Hosting Bundle 安裝順序很重要！
Hosting Bundle 必須在 IIS 功能啟用之後安裝。若先裝 Hosting Bundle 再啟用 IIS，模組不會自動註冊。 若已發生此情況，到「控制台 → 程式與功能」找到 Hosting Bundle，選「修復（Repair）」後再執行 iisreset。
💡
安裝任一模組後都要執行 iisreset 讓 IIS 重新載入模組清單。

第零步：安裝 ARR 並啟用 Proxy

ARR（Application Request Routing）負責將前端的 /api/* 請求轉發到本機 API，是讓 CRUD 功能正常運作的關鍵。

0-1　下載並安裝 ARR 3.0

1
前往 https://www.iis.net/downloads/microsoft/application-request-routing
下載「Application Request Routing 3.0」→ 檔名：requestRouter_amd64.msi（約 9 MB）
2
以系統管理員身份執行安裝程式，一路 Next 完成。
3
重新開啟 IIS 管理員（不需重開機）。
4
點選左側最上層伺服器節點（非網站），確認首頁 IIS 區塊出現「Application Request Routing Cache」圖示。
💡
此圖示只出現在伺服器頂層節點的功能檢視，點到網站節點（如 kmsh-whiteboard）會看不到。

0-2　啟用 Proxy 功能

ARR 安裝後，proxy 功能預設是關閉的，需手動開啟一次。

1
雙擊「Application Request Routing Cache」。
2
右側「動作」面板點「Server Proxy Settings...」。
3
勾選「Enable proxy」→ 按右側 Apply。
✅
驗證：瀏覽器開 http://localhost/api/Text，應看到 []（與直接開 http://localhost:5100/api/Text 結果相同）。 若成功，TestPage 頁面在載入時就不會再出現 IIS 錯誤代碼閃現的情形。

第二步：部署 React 前端到 IIS

2-1　Build React 專案

cd "C:\WorkDir\Nursing Whiteboard\React"
npm run build

產出目錄：C:\WorkDir\Nursing Whiteboard\React\dist\
其中包含 web.config（已預先放在 public/ 目錄，build 時自動複製進來）。

2-2　複製到 IIS 目錄

New-Item -ItemType Directory -Path "C:\inetpub\wwwroot\kmsh-whiteboard" -Force
Copy-Item -Path "C:\WorkDir\Nursing Whiteboard\React\dist\*" `
          -Destination "C:\inetpub\wwwroot\kmsh-whiteboard" `
          -Recurse -Force

複製完成後，C:\inetpub\wwwroot\kmsh-whiteboard\ 應包含：

kmsh-whiteboard\
  assets\
  favicon.svg
  icons.svg
  index.html
  web.config     ← 這個一定要在！

2-3　web.config 說明（自動產生，不需手動建立）

React\public\web.config 已預先設定好兩條規則，build 後自動出現在 dist 裡：

<!-- 規則一：/api/* 轉發到本機 API（port 5100） -->
<rule name="API Proxy" stopProcessing="true">
  <match url="^api/(.*)" />
  <action type="Rewrite" url="http://localhost:5100/api/{R:1}" />
</rule>

<!-- 規則二：所有非實體檔案路由 fallback 到 index.html -->
<rule name="SPA Fallback" stopProcessing="true">
  <match url=".*" />
  <conditions logicalGrouping="MatchAll">
    <add input="{REQUEST_FILENAME}" matchType="IsFile" negate="true" />
    <add input="{REQUEST_FILENAME}" matchType="IsDirectory" negate="true" />
  </conditions>
  <action type="Rewrite" url="/index.html" />
</rule>

第三步：防火牆開放 Port 80

New-NetFirewallRule -DisplayName "IIS kmsh-whiteboard HTTP" `
                   -Direction Inbound -Protocol TCP `
                   -LocalPort 80 -Action Allow

此指令需以系統管理員身份執行。若規則已存在可略過。

日後更新方式

更新 React 前端

cd "C:\WorkDir\Nursing Whiteboard\React"
npm run build
Copy-Item -Path ".\dist\*" -Destination "C:\inetpub\wwwroot\kmsh-whiteboard" -Recurse -Force

不需重啟 IIS，靜態檔案即時生效。

更新 .NET API

cd "C:\WorkDir\Nursing Whiteboard\API\kmsh-whiteboard"
dotnet publish -c Release -o "C:\inetpub\wwwroot\kmsh-api"
💡
publish 時 IIS 若鎖住 dll 無法覆蓋，先在 IIS 管理員停止 kmsh-whiteboard-api 網站，publish 完再啟動。 或啟用 app_offline.htm：在 C:\inetpub\wwwroot\kmsh-api\ 建立一個空的 app_offline.htm，IIS 會自動卸載應用程式，讓 publish 完成；完成後刪除此檔即可恢復。

常見錯誤排查

HTTP 500.19 — 錯誤碼 0x8007000d（API web.config）
原因：IIS 找不到 AspNetCoreModuleV2，代表 .NET 8 Hosting Bundle 未正確安裝到 IIS。
解法：
  1. 從 dotnet.microsoft.com/download/dotnet/8.0 下載 Hosting Bundle（非 x64 Runtime）
  2. 以系統管理員執行安裝程式
  3. 執行 iisreset
  4. 確認：Get-WebGlobalModule | Where-Object { $_.Name -like "*AspNetCore*" } 有輸出
⚠️ 若已安裝仍出錯，可能是 IIS 啟用在 Hosting Bundle 之後。到控制台找到 Hosting Bundle → 選「修復」。
HTTP 500.19 — 錯誤碼 0x8007000d（前端 web.config）
原因：<rewrite> 標籤需要 URL Rewrite 模組，模組未安裝時 IIS 無法識別此設定。
解法：
  1. 下載 URL Rewrite：iis.net/downloads/microsoft/url-rewrite
  2. 安裝 rewrite_amd64_en-US.msi
  3. 執行 iisreset
/test 直接輸入網址回 IIS 404
原因：URL Rewrite 的 SPA Fallback 規則未生效，或 web.config 不在正確位置。
解法： 確認 C:\inetpub\wwwroot\kmsh-whiteboard\web.config 存在。若不存在，重新執行 npm run build 並複製 dist。
TestPage 載入時出現一堆 HTML 代碼並在 3 秒後消失
原因：TestPage 載入時自動呼叫 GET /api/Text，但 IIS 回傳 404 HTML 錯誤頁（ARR 未安裝或 Proxy 未啟用），錯誤訊息把整段 HTML 顯示出來，3 秒後 setTimeout 自動清除。
解法：
  1. 確認 IIS 管理員伺服器頂層節點首頁有「Application Request Routing Cache」圖示
    → 若沒有：下載安裝 requestRouter_amd64.msi，重新開啟 IIS 管理員
  2. 雙擊「Application Request Routing Cache」→ Server Proxy Settings → Enable proxy 已勾選
  3. 驗證：瀏覽器開 http://localhost/api/Text 應回傳 []
/api/Text 回 502 Bad Gateway
原因：ARR Proxy 已啟用但 API 網站未在 port 5100 執行，或 port 設定錯誤。
解法：
  1. 確認 http://localhost:5100/api/Text 本機可直接訪問
  2. 確認 IIS kmsh-whiteboard-api 網站狀態為「已啟動」
  3. 確認 API 網站繫結為 127.0.0.1:5100（非 0.0.0.0）
API 更新時 publish 失敗（檔案被鎖定）
原因：IIS 的 worker process 正在使用 dll，無法覆蓋。
解法： 在 C:\inetpub\wwwroot\kmsh-api\ 建立空白的 app_offline.htm， IIS 會自動卸載應用程式。publish 完成後刪除此檔即可。
